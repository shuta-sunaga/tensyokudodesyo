# Lark Base 連携セットアップ手順

「転職どうでしょう」のお問い合わせフォームから送信された内容を Lark Base に格納するために必要なセットアップ手順です。**管理者権限が必要な作業のみ** をまとめています。

このドキュメントの作業が完了したら、最後の「開発者へ渡す情報」セクションに記載した6項目を開発者へお伝えください。

---

## 全体像

```
[ユーザー] → [/contact/ フォーム]
              ↓ POST
         [AWS Lambda]
              ├─→ Lark Base API (レコード追加)  ← このドキュメントで設定するもの
              └─→ AWS SES (確認メール送信)
```

Lambda が Lark Base API を呼ぶには、以下の3つが必要です：

1. **Lark カスタムアプリ**（App ID / App Secret を発行する）
2. **Base（多次元表）と Table**（受信箱）
3. **アプリへの権限付与**（Base への書き込み権限）

---

## Step 1: Lark Open Platform でアプリを作成

### 1-1. ログイン

https://open.larksuite.com/app にアクセスし、組織管理者アカウントでログインします。

### 1-2. カスタムアプリを作成

1. 上部メニューの「**Create App**」→「**Custom App**」を選択
2. 入力項目:
   - **App Name**: `転職どうでしょう - お問い合わせフォーム`
   - **App Description**: `tensyokudodesyo.com のお問い合わせフォームから Lark Base にレコードを書き込む内部アプリ`
   - **App Icon**: ロゴ画像（任意・後から変更可）
3. 「Create」を押す

### 1-3. App ID / App Secret を控える

作成完了画面または「**Credentials & Basic Info**」タブに表示されます：

- **App ID**: `cli_xxxxxxxxxxxxxxxx`（実値はこのドキュメントには書かない）
- **App Secret**: `XXXXXXXXXXXXXXXXXXXXXXXXXXX`（実値はこのドキュメントには書かない）

> ⚠️ **このドキュメントは Git 管理されるためにあるプレースホルダです。実際の値は絶対にここに記入しないでください。** 開発者には 1Password などの暗号化チャネルでお渡しください。

⚠️ **App Secret は秘密情報です**。Slack/メール本文等の公開チャネルで共有しないでください。後述のとおり 1Password やパスワードマネージャー経由で開発者にお渡しください。

### 1-4. 権限スコープ追加

左メニュー「**Permissions & Scopes**」→「**Add Scopes**」から、以下4つを追加します：

| Permission Code | 説明 |
|----------------|------|
| `bitable:app` | Base アプリへのアクセス |
| `bitable:app:readonly` | 上記の読み取り(`bitable:app` に内包) |
| 不要なら省略可: `im:message` | 通知用（将来用、今回は不要） |
| 不要なら省略可: `contact:user.id:readonly` | 担当者の特定（今回は不要） |

最低限必要なのは **`bitable:app`** のみです。

「**Apply for Release**」ボタンが出る場合は、**カスタムアプリは内部利用なのでリリース申請は不要**です。「Save Draft」のみで進めて構いません。

### 1-5. アプリを「組織内で利用可能」にする

左メニュー「**Visibility**」→「Visible to: Specific members」または「All members of the organization」を選択。

開発者が API を叩く際に必要です。

## Step 2: Lark Base（多次元表）を作成

Lark の任意のフォルダで「**新規作成**」→「**Base**」を選びます。

### 2-1. Base 名と Table 名

- **Base 名**: `転職どうでしょう - お問い合わせ`
- **Table 名（最初のシート名）**: `submissions`

### 2-2. テーブルのフィールド設計

以下のフィールドを **この順番** で作ります。フィールド名は **完全一致** が必要です（API キーになるため）。

| 番号 | フィールド名 | タイプ | 必須 | 備考 |
|------|------------|--------|------|------|
| 1 | `submitted_at` | **日時** | ✅ | 受信日時（自動入力ではなくAPIから設定） |
| 2 | `name` | テキスト | ✅ | 氏名 |
| 3 | `name_kana` | テキスト | | ふりがな（任意） |
| 4 | `email` | テキスト | ✅ | メールアドレス |
| 5 | `phone` | テキスト | | 電話番号（任意） |
| 6 | `prefecture` | 単一選択 | ✅ | 47都道府県（下記参照） |
| 7 | `age` | 数値 | ✅ | 年齢（18-80） |
| 8 | `current_status` | 単一選択 | | 在職中/離職中/転職活動中/情報収集中 |
| 9 | `interested_job` | テキスト | | 気になった求人(URL or 名称) |
| 10 | `message` | テキスト（複数行） | | ご質問・ご要望 |
| 11 | `terms_consent` | チェックボックス | ✅ | 利用規約同意 |
| 12 | `terms_version` | テキスト | ✅ | 同意した利用規約の版数（例: `2026-05-07`） |
| 13 | `privacy_consent` | チェックボックス | ✅ | 個人情報保護方針同意 |
| 14 | `privacy_policy_version` | テキスト | ✅ | 同意したポリシーの版数（例: `2026-04-06`） |
| 15 | `consent_ip` | テキスト | ✅ | 同意時の IP アドレス（証跡用） |
| 16 | `user_agent` | テキスト | | ブラウザUA |
| 17 | `referrer_url` | テキスト | | 流入元URL |
| 18 | `status` | 単一選択 | | デフォルト「未対応」 / 値: 未対応, 対応中, 完了, 不要対応 |
| 19 | `notes` | テキスト（複数行） | | 運用メモ |

> ⚠️ **既に Base を作成済みの方へ**: `terms_consent` (チェックボックス) と `terms_version` (テキスト) の2フィールドを既存テーブルに**追加**してください。フィールド名は完全一致で。

### 2-4. App Token と Table ID を控える

#### App Token（= Base ID）

Base 画面の URL を見ます：

```
https://xxxxxx.jp.larksuite.com/base/{APP_TOKEN}?table={TABLE_ID}&view=...
```

`/base/` の後ろの文字列が **App Token** です（例: `bascn1234567890abcdef`）。

#### Table ID

URL の `?table=` の後ろの値（例: `tblXXXXXXXXXXXX`）が **Table ID** です。

両方をメモしておきます。

## Step 3: アプリに Base への権限付与

カスタムアプリは作っただけでは特定 Base にアクセスできません。Base 側で明示的に招待します。

### 3-1. Base を開く

Step 2 で作成した「転職どうでしょう - お問い合わせ」Base を開きます。

### 3-2. アプリを共有

1. 右上の「**...** (More)」→「**Add Documents Apps**」または「**Permission**」
2. 検索ボックスにアプリ名「転職どうでしょう - お問い合わせフォーム」を入力
3. 表示されたアプリを選択
4. 権限を **「編集可能」(Can Edit)** に設定
5. 確定

### 3-3. 動作確認（任意）

開発者側で curl テストすれば確認できますが、ここでは Step 5 でまとめてご確認いただきます。

---

## Step 4: 受信通知（任意）

新規お問い合わせを Lark のチャットへ自動通知したい場合：

### Option A: Lark Base の自動化

1. Base の右上「**Automation**」を開く
2. 「**Create Workflow**」→ Trigger: `When record is added`
3. Action: `Send notification` → 担当者または共有チャットを指定
4. メッセージ例:
   ```
   新しいお問い合わせが届きました。
   氏名: {name}
   都道府県: {prefecture}
   気になった求人: {interested_job}
   ```

### Option B: Webhook + Lark チャット
今回のスコープ外。必要なら別途依頼。

## Step 5: 開発者へ渡す情報

以下6項目を **1Password の安全な共有リンク** または **Lark の社内チャット (DM)** で開発者にお渡しください。

| # | 項目 | 取得方法 |
|---|------|---------|
| 1 | App ID | Step 1-3 |
| 2 | App Secret | Step 1-3（**秘密情報・口頭/暗号化チャネルで**） |
| 3 | App Token (Base ID) | Step 2-4 |
| 4 | Table ID | Step 2-4 |
| 5 | テナントドメイン | URLの `xxxxxx.jp.larksuite.com` の `xxxxxx` 部分 |
| 6 | 通知用チャネル ID（任意） | Step 4 で自動化を組まない場合は不要 |

開発者はこれらを **AWS Secrets Manager** に格納し、Lambda が読み取る形にします。**`.env` やコードには絶対に書き込みません**。

---

## Step 6: 動作確認（開発者作業後）

開発者から「ローカルで送信テストできました」と連絡が来たら：

1. Base を開いて、テストレコードが入っているか確認
2. レコードのフィールド内容が正しいか確認（特に `submitted_at` が現地時刻か、`prefecture` が選択肢で表示されているか）
3. 件数増加・編集権限が問題ないか確認

問題なければ次フェーズ（本番デプロイ）へ進みます。

## トラブルシューティング

| 症状 | 原因 | 対処 |
|------|------|------|
| `99991663` (No permission) | アプリが Base に招待されていない | Step 3 を再実施 |
| `1254003` (Field not found) | フィールド名のtypoまたは未作成 | Step 2 のフィールド名を再確認（小文字スネークケース） |
| `1254044` (Table not found) | Table ID 誤り | URL から再取得 |
| 単一選択の値が反映されない | 選択肢に存在しない値を送信 | Step 2-3 の選択肢を再確認 |

---

## セキュリティ上の注意

- **App Secret** は AWS Secrets Manager に格納し、Lambda 環境変数に **直接書かない**
- Base のアクセス権限は最小限のメンバーに（運営担当のみ）
- 個人情報を含むレコードを **Lark の他チャネル（チャット等）に転載しない**
- 退職者が出たら、その人の Base アクセス権限を削除する
- 月1回、Base のアクセス監査ログを確認する
